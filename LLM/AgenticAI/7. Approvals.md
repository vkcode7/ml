### 1. MRMP — Model Risk Management Policy

The **foundational governance document** that defines rules every AI/model must satisfy before approval.

```
MRMP FRAMEWORK
──────────────────────────────────────────────────────
Purpose:
  Establish the firm's risk appetite, validation
  requirements, and lifecycle controls for all models.

Key Components:
┌─────────────────────────────────────────────────────┐
│ 1. Model Definition & Scope                         │
│    - What qualifies as a "model" under policy       │
│    - LLMs, ML models, statistical models, hybrids   │
│    - Exclusions (simple calculations, rule engines) │
│                                                     │
│ 2. Model Tiering                                    │
│    ├── Tier 1: High risk (trading, credit, FO AI)   │
│    ├── Tier 2: Medium risk (analytics, reporting)   │
│    └── Tier 3: Low risk (internal tooling)          │
│                                                     │
│ 3. Mandatory Approval Gates per Tier                │
│    Tier 1: DEPQ + MRC + CRO sign-off               │
│    Tier 2: MRC + Risk sign-off                      │
│    Tier 3: Peer review + manager sign-off           │
│                                                     │
│ 4. Validation Independence Requirement              │
│    - Validators MUST be separate from developers    │
│    - No conflict of interest                        │
│    - Documented evidence required                   │
│                                                     │
│ 5. Model Lifecycle Requirements                     │
│    - Development → Validation → Approval            │
│    → Deployment → Monitoring → Retirement           │
│                                                     │
│ 6. Regulatory Alignment                             │
│    - SR 11-7 (Federal Reserve)                      │
│    - OCC 2011-12                                    │
│    - EU AI Act                                      │
│    - Basel III/IV                                   │
└─────────────────────────────────────────────────────┘

Key Approval it triggers:
  - Every model must be registered in the Model Inventory
  - Tier 1 models cannot deploy without full MRMP compliance
  - Annual review mandatory regardless of changes
```

### 2. DEPQ — Data, Ethics, Privacy & Quality

The **data governance approval** — proves the model's data is clean, ethical, private, and fit for purpose.

```
DEPQ APPROVAL FRAMEWORK
──────────────────────────────────────────────────────
Purpose:
  Ensure AI models use data that is ethically sourced,
  privacy-compliant, high quality, and lineage-tracked.

4 Pillars:

┌─────────────────────────────────────────────────────┐
│ D — DATA LINEAGE & QUALITY                          │
│                                                     │
│ Approvals required:                                 │
│  ✅ Data source inventory documented                │
│  ✅ Data quality scores meet thresholds             │
│     (completeness, accuracy, timeliness > 95%)      │
│  ✅ Data lineage diagram approved                   │
│  ✅ Data dictionary signed off                      │
│  ✅ No unauthorized data sources                    │
│  ✅ Training/test data split documented             │
│  ✅ Data retention policy compliant                 │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│ E — ETHICS & FAIRNESS                               │
│                                                     │
│ Approvals required:                                 │
│  ✅ Bias testing completed (all protected groups)   │
│  ✅ Fairness metrics within threshold (< 5% delta)  │
│  ✅ Ethics review board sign-off                    │
│  ✅ No discriminatory outcomes demonstrated         │
│  ✅ Human oversight in place for consequential      │
│     decisions (GDPR Art. 22 compliance)             │
│  ✅ AI use case ethical risk assessment complete    │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│ P — PRIVACY                                         │
│                                                     │
│ Approvals required:                                 │
│  ✅ Privacy Impact Assessment (PIA) completed       │
│  ✅ PII inventory documented                        │
│  ✅ Data minimization principle applied             │
│  ✅ Consent mechanisms in place (if applicable)     │
│  ✅ Cross-border data transfer restrictions met     │
│  ✅ Right to erasure process defined                │
│  ✅ DPO (Data Protection Officer) sign-off          │
│                                                     │
│ Regulatory alignment:                               │
│  - GDPR (EU)                                        │
│  - CCPA (California)                                │
│  - GLBA (US financial data)                         │
│  - FCA Data Standards (UK)                          │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│ Q — QUALITY ASSURANCE                               │
│                                                     │
│ Approvals required:                                 │
│  ✅ Model performance meets acceptance criteria     │
│  ✅ Backtesting report signed off                   │
│  ✅ Stress & sensitivity tests passed               │
│  ✅ Regression test suite in CI/CD pipeline         │
│  ✅ Independent QA review complete                  │
│  ✅ Defect log reviewed — no critical open issues   │
│  ✅ UAT (User Acceptance Testing) signed off        │
│     by business stakeholder                         │
└─────────────────────────────────────────────────────┘
```

### 3. The Full Approval Chain

```
STAGE 1: INITIATION
──────────────────────────────────────────────────────
  □ Model Registration
    - Register in firm's Model Inventory
    - Assign Model ID, Tier, Owner
    - Notify Model Risk Management

  □ Use Case Approval
    - Business sponsor submits use case brief
    - AI Ethics Committee reviews
    - Legal reviews regulatory permissibility
    - CIO/CTO signs off on technology approach


STAGE 2: DEVELOPMENT APPROVALS
──────────────────────────────────────────────────────
  □ DEPQ — Data Approval
    - Data lineage & quality sign-off
    - Privacy Impact Assessment approved
    - Ethics & bias framework agreed
    - DPO sign-off on data usage

  □ Architecture Review Board (ARB)
    - Technology stack approved
    - Security architecture reviewed
    - Cloud/infra design approved
    - Integration patterns signed off

  □ Information Security Review
    - Penetration testing (adversarial/prompt injection)
    - Data encryption standards met
    - Access controls reviewed
    - Third-party AI vendor risk assessed
      (e.g. OpenAI, Anthropic vendor risk review)


STAGE 3: VALIDATION APPROVALS
──────────────────────────────────────────────────────
  □ Independent Model Validation (MRMP requirement)
    - Validation team confirms independence
    - All validation artifacts completed
    - Validation report issued with findings
    - All critical findings remediated

  □ Model Risk Committee (MRC) Pre-Review
    - Preliminary findings presented
    - Material issues flagged early
    - Conditional approvals documented


STAGE 4: COMPLIANCE & LEGAL SIGN-OFFS
──────────────────────────────────────────────────────
  □ Compliance Approval
    - Regulatory mapping complete
      (SR 11-7, EU AI Act, GDPR, FINRA, MiFID II)
    - Explainability requirements met
    - Consumer protection rules satisfied
    - Regulatory notification filed (if required)

  □ Legal Review
    - IP ownership of model outputs clarified
    - Third-party AI vendor contract reviewed
    - Liability framework documented
    - Client disclosure requirements met

  □ Operational Risk Sign-off
    - Business continuity plan for model failure
    - Fallback/failover procedures documented
    - Incident response plan in place


STAGE 5: FINAL APPROVAL GATES
──────────────────────────────────────────────────────
  □ MRC — Model Risk Committee
    ├── Quorum: CRO, Head of Model Risk,
    │           Compliance, Business Sponsor,
    │           Independent Validator
    ├── Reviews full artifact package
    ├── Votes: Approve / Conditional / Reject
    └── Conditions must be resolved before deploy

  □ CRO Sign-off (Tier 1 models)
    - Chief Risk Officer final approval
    - Risk appetite confirmed
    - Residual risks accepted

  □ Change Advisory Board (CAB)
    - Production deployment approved
    - Rollback plan reviewed
    - Deployment window assigned


STAGE 6: POST-DEPLOYMENT
──────────────────────────────────────────────────────
  □ Hypercare Review (30/60/90 days)
    - Live performance vs validation benchmarks
    - Drift monitoring active
    - Issues remediated within SLA

  □ Annual Revalidation (MRMP requirement)
    - Full revalidation every 12 months
    - Triggered earlier by material changes
    - MRC re-approval required
```

### 4. Regulatory Frameworks Driving These Approvals

| Regulation | Jurisdiction | What it mandates |
|---|---|---|
| **SR 11-7** | US Federal Reserve | Independent validation, documentation, governance for all models |
| **OCC 2011-12** | US OCC | Model risk management for national banks |
| **EU AI Act** | European Union | High-risk AI conformity assessment, human oversight, transparency |
| **GDPR Art. 22** | EU | No fully automated decisions on individuals without human review |
| **MiFID II** | EU | Algorithmic trading controls, audit trails |
| **FINRA Rule 3110** | US | Supervisory systems for AI-assisted advice |
| **BCBS 239** | Basel Committee | Data quality and risk data aggregation |
| **FCA PS22/3** | UK | Consumer duty — AI must deliver good client outcomes |

### 6. Common Approval Failure Points

| Failure | Why it happens | How to avoid |
|---|---|---|
| **Validation not independent** | Developer validates own model | Separate reporting lines from day 1 |
| **DEPQ data gaps** | Undocumented data sources found late | Data inventory before Sprint 1 |
| **Bias test failures** | Protected group analysis skipped | Include in Sprint 3 acceptance criteria |
| **MRC package incomplete** | Artifacts produced last-minute | Artifact checklist tracked each sprint |
| **Vendor risk not assessed** | OpenAI/Anthropic not in procurement | Third-party AI vendor review at initiation |
| **Explainability gap** | Model can't explain decisions | Chain-of-thought logging from day 1 |
| **No rollback plan** | CAB blocks deployment | BCP documented before Stage 5 |

---

**One-liner summary:**
> "In financial institutions, AI model approval isn't a single sign-off — it's a staged governance chain where MRMP sets the rules, DEPQ proves your data is clean and ethical, and the MRC is the final court that weighs every artifact before anything touches production."
