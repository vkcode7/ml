# Tool Development & Integration: 
Build and integrate APIs/tools, enabling secure and reliable agent access to internal and external resources.

### Introduction to Tool Development & Integration

In AI-driven automation, tools are the "hands" of your agents—extensions that allow them to interact with the real world beyond just generating text. Tool development and integration involve creating or connecting APIs, databases, services, or custom functions so agents can access resources securely and reliably. This enables agents to perform actions like querying a CRM system, sending emails, or integrating with third-party services (e.g., weather APIs for logistics agents).

This lesson builds on agent design by focusing on the "action" component. We'll cover concepts, step-by-step building, integration techniques, security best practices, and deployment. Frameworks like LangChain, LlamaIndex, or Haystack make this easier, but we'll emphasize custom development for business needs. By the end, you'll know how to empower agents to handle real workflows, such as a sales agent pulling customer data from Salesforce while complying with data privacy laws.

### 1. Core Concepts
Understand the basics before building:

- **What are Tools in AI Agents?**
  Tools are callable functions or APIs that agents invoke based on reasoning. They bridge the gap between an LLM's knowledge and external systems. Examples: A database query tool for internal inventory, or an external API for stock prices.

- **Types of Tools:**
  - **Internal Tools:** Access company resources like databases (SQL/NoSQL), intranets, or proprietary software.
  - **External Tools:** Integrate with public APIs (e.g., Google Maps, Stripe) or third-party services.
  - **Custom Tools:** Built from scratch, like a Python function for data analysis.
  - **Hybrid Tools:** Combine multiple (e.g., fetch weather externally, then update internal logistics DB).

- **Integration Goals:**
  - **Security:** Prevent unauthorized access, data leaks, or injections.
  - **Reliability:** Handle errors, rate limits, and downtime gracefully.
  - **Scalability:** Support high-volume calls without bottlenecks.
  - **Agent Compatibility:** Tools must be describable in natural language for LLMs to "understand" when to use them (e.g., via tool schemas).

- **Challenges in Business Contexts:**
  - Alignment with objectives: Tools should support goals like efficiency without introducing risks.
  - Regulatory Compliance: Ensure tools handle sensitive data per laws (e.g., PCI-DSS for payments).

Example: An HR agent uses a tool to integrate with LinkedIn API (external) to fetch candidate profiles, then stores them in an internal ATS (applicant tracking system) while anonymizing PII for GDPR.

### 2. Building Tools
Start by designing tools that are modular and reusable.

#### Step 2.1: Define Tool Requirements
- Analyze the workflow: What data/actions does the agent need? (e.g., For a customer support agent: Read from Zendesk, write to Slack.)
- Specify inputs/outputs: Use schemas like JSON for structured data.
- Consider constraints: Rate limits, authentication needs, error types.

#### Step 2.2: Develop the Tool
- **Choose a Language/Framework:**
  - Python: Ideal for rapid development with libraries like Requests for APIs.
  - Node.js: For real-time integrations.
  - Use wrappers: LangChain's `Tool` class simplifies creation.

- **Build a Basic Tool:**
  Example: A custom Python tool to query an internal database.
  ```python
  from langchain.tools import BaseTool
  import sqlite3  # Or your DB library

  class DatabaseQueryTool(BaseTool):
      name = "database_query"
      description = "Query the internal company database for employee info. Input: SQL query string."

      def _run(self, query: str):
          conn = sqlite3.connect('company.db')
          cursor = conn.cursor()
          try:
              cursor.execute(query)
              results = cursor.fetchall()
              return results
          except Exception as e:
              return f"Error: {str(e)}"
          finally:
              conn.close()
  ```

- **For APIs:** Use HTTP clients.
  Example: External weather tool.
  ```python
  import requests

  class WeatherTool(BaseTool):
      name = "get_weather"
      description = "Get current weather for a city. Input: City name."

      def _run(self, city: str):
          api_key = "your_openweather_api_key"  # Securely stored
          url = f"https://api.openweathermap.org/data/2.5/weather?q={city}&appid={api_key}"
          response = requests.get(url)
          if response.status_code == 200:
              return response.json()
          else:
              raise ValueError("API request failed")
  ```

Tip: Always validate inputs to prevent SQL injections or invalid API calls.

#### Step 2.3: Test the Tool Standalone
- Unit tests: Use pytest to simulate inputs/outputs.
- Edge cases: Test failures like network errors or invalid data.

### 3. Integrating Tools with Agents
Once built, connect tools to agents for autonomous use.

#### Step 3.1: Register Tools
- In frameworks:
  - LangChain: `agent = initialize_agent(tools=[DatabaseQueryTool(), WeatherTool()], llm=your_llm)`
  - CrewAI: Define tasks that reference tools.
  - AutoGen: Use `register_function` to expose tools.

- Provide Descriptions: LLMs decide tool usage based on natural-language descriptions. Make them clear and concise.

#### Step 3.2: Enable Agent Access
- **Prompt Engineering:** Instruct agents on tool usage (e.g., "Use get_weather only for location-based queries").
- **Dynamic Tool Selection:** Use reasoning chains (e.g., ReAct: Reason + Act) where agents think step-by-step.
- **Multi-Tool Orchestration:** For complex flows, chain tools (e.g., query DB → analyze with custom function → update external API).

Example Integration:
- Agent Prompt: "You are a logistics agent. Use tools to check inventory and weather before scheduling shipments."
- Workflow: Agent reasons: "Need inventory? Use database_query. Need forecast? Use get_weather."

#### Step 3.3: Handle Asynchronous and Parallel Calls
- Use async libraries (e.g., aiohttp) for non-blocking calls.
- Parallelism: In multi-agent setups, run tools concurrently to speed up workflows.

### 4. Ensuring Security and Reliability
Critical for production, especially with sensitive resources.

#### Step 4.1: Security Best Practices
- **Authentication:** Use OAuth2, API keys, or JWT. Store secrets in env vars or vaults (e.g., AWS Secrets Manager).
- **Authorization:** Role-based access (e.g., agent can read but not write to DB).
- **Input Sanitization:** Validate/escape inputs to avoid injections.
- **Encryption:** TLS for external calls; encrypt data at rest.
- **Least Privilege:** Agents get minimal access (e.g., read-only DB views).
- **Compliance:** Log accesses for audits; use tools like OPA (Open Policy Agent) for policy enforcement.

#### Step 4.2: Reliability Measures
- **Error Handling:** Retry logic (e.g., exponential backoff with libraries like Tenacity).
- **Fallbacks:** If tool fails, default to human escalation or alternative tool.
- **Monitoring:** Track usage with Prometheus/Grafana; alert on failures.
- **Rate Limiting:** Implement client-side throttling to respect API limits.
- **Versioning:** Use semantic versioning for tools to avoid breaking changes.

Common Pitfall: Over-reliance on external tools—have redundancies (e.g., cache results).

### 5. Deployment and Scaling
- **Infrastructure:** Host tools as microservices (e.g., Flask/Django APIs) on Kubernetes for scalability.
- **CI/CD:** Use GitHub Actions to automate testing/integration.
- **Cloud Integration:** AWS Lambda for serverless tools; API Gateway for exposure.
- **Testing in Production:** Use canary deployments; monitor with tools like Sentry.

# Agent Memory and State Management

**Agent Memory and State Management** is one of the most critical (and often under-engineered) aspects of building reliable, autonomous AI agents in production. Without good memory, agents forget context mid-task, repeat mistakes, lose user preferences, or fail to improve over time. In business workflows, this directly impacts efficiency, compliance (audit trails), personalization, and cost (avoiding redundant LLM calls).

This lesson explains the concepts, types, patterns, implementation strategies (focusing on 2025–2026 realities), and best practices. We'll cover single-agent and multi-agent scenarios.

### 1. Core Concepts: Memory vs. State
- **State** — The current snapshot of everything needed to continue execution: messages, tool outputs, variables, decisions, progress in a workflow. In graph-based systems it's often a typed object/dict.
- **Memory** — The mechanism to persist and retrieve relevant information **across steps, turns, or even sessions**. Memory is usually a subset or derived view of state history.

Key distinction in 2026-era frameworks:
- Short-lived / in-context → part of prompt (short-term)
- Persistent / retrievable → stored externally (long-term)

### 2. Types of Memory in Modern AI Agents
Agents borrow from human cognition but adapt to LLM constraints (token limits, cost, latency).

| Type              | Duration       | What it Stores                              | Typical Implementation                  | Use Case Example                              | Frameworks with Native Support |
|-------------------|----------------|---------------------------------------------|-----------------------------------------|-----------------------------------------------|--------------------------------|
| Short-term / Working / In-context | Current session / few turns | Recent messages, current plan, intermediate results | LLM context window, message list       | Remembering the last 3–5 user utterances     | All (LangChain RunnableWithMessageHistory, CrewAI short-term) |
| Long-term (general) | Across sessions | User facts, preferences, domain knowledge  | Vector DB + RAG, key-value store       | "User prefers concise reports"               | Mem0, LangGraph + vector store, Redis |
| Episodic          | Across sessions | Specific past events/experiences            | Timestamped summaries or full traces   | "Last time we tried vendor X, delivery failed" | LangGraph checkpoints + summarization, Mem0 episodic |
| Semantic          | Persistent     | Facts, concepts, relationships              | Knowledge graph / vector embeddings    | Company org chart, product catalog           | Vector DBs (Pinecone, Weaviate), Knowledge Graphs |
| Procedural        | Persistent     | How-to knowledge (skills, patterns)         | Fine-tuned models, tool libraries, few-shot examples | Improved prompting after seeing failures     | Rarely native; often RLHF or tool reflection loops |

Most production agents in 2025–2026 use **hybrid** approaches: short-term in prompt + long-term via retrieval.

### 3. State Management Patterns (2026 Perspective)
State management has evolved significantly. The dominant paradigm is **graph + checkpointing**.

#### Pattern A: Simple Conversation Memory (Single-turn / Chat)
Use when the agent is mostly conversational.

```python
# LangChain 0.3+ style (2025–2026 recommended)
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_core.messages import AIMessage, HumanMessage

# Memory backend: in-memory, Redis, Postgres, etc.
store = {}  # or RedisSaver, PostgresSaver

def get_session_history(session_id: str):
    if session_id not in store:
        store[session_id] = []
    return store[session_id]

chain = prompt | llm | parser
with_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",
    history_messages_key="chat_history"
)

# Thread-safe session
config = {"configurable": {"session_id": "user_123"}}
```

#### Pattern B: Graph State + Checkpointing (LangGraph — de-facto standard for complex agents)
LangGraph treats the agent as a state machine. State is explicit, versioned, and check-pointed.

```python
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.memory import MemorySaver  # or PostgresSaver
from typing import TypedDict, Annotated
import operator

class AgentState(TypedDict):
    messages: Annotated[list, operator.add]     # short-term
    plan: str                                   # current plan
    findings: list                              # episodic accumulation
    user_preferences: dict                      # long-term (merged)

graph = StateGraph(state_schema=AgentState)

# Nodes update state immutably
def researcher(state):
    # ... use tools ...
    return {"findings": [...], "messages": [AIMessage(...)]}

# Checkpointing = durable memory
checkpointer = MemorySaver()   # swap to Redis/Postgres for prod
app = graph.compile(checkpointer=checkpointer)

# Run with thread_id = persistent session key
app.invoke(inputs, config={"configurable": {"thread_id": "task_abc123"}})
```

Advantages in 2026:
- Human-in-the-loop breakpoints
- Time-travel debugging (replay from any checkpoint)
- Resume after crash / server restart

#### Pattern C: Multi-Agent Memory (CrewAI, AutoGen, LangGraph multi-agent)
- **CrewAI**: Each agent has role-based short/long-term memory + shared crew memory. Entity memory tracks people/objects.
- **AutoGen**: Primarily message-based (conversation history passed around). Can add external memory via group chat manager.
- **LangGraph**: Most flexible — shared state object + per-node memory + background summarization.

Best practice for multi-agent in 2026: Use **shared long-term memory** (vector DB or Redis) + **per-agent episodic scratchpad** + **crew/global state** for coordination.

### 4. Implementation Best Practices (Production 2026)
1. **Separate hot-path from cold-path memory**
   - Hot: Short-term + recent episodic → in prompt (fast)
   - Cold: Summarized long-term + semantic → retrieved on demand (RAG)

2. **Intelligent retrieval & summarization**
   - Use MMR (maximal marginal relevance) or HyDE to avoid redundant context
   - Background summarizer job: condense old messages → episodic summaries

3. **Persistence choices**
   - In-memory (dev) → MemorySaver
   - Fast & durable → Redis (checkpoints + vector)
   - Audit-heavy → PostgreSQL + immutable logs

4. **Forgetting & cleanup**
   - TTL on short-term entries
   - Decay / importance scoring on long-term items
   - User-controlled forget (GDPR "right to be forgotten")

5. **Consistency & conflict resolution**
   - Versioned facts (last write wins or merge logic)
   - Conflict detector node before critical actions

6. **Observability**
   - LangSmith / Phoenix for tracing memory injections
   - Log what was retrieved vs. hallucinated

### 5. Quick Comparison Table — Popular Frameworks (2026 View)

| Framework   | Short-term     | Long-term / Persistent | State Style          | Multi-agent Memory | Production Strength |
|-------------|----------------|------------------------|----------------------|--------------------|---------------------|
| LangGraph   | Messages in state | Checkpoint + custom | Explicit TypedDict  | Excellent (shared state) | Highest (durability, HITL) |
| CrewAI      | Built-in short/long | Entity + role memory  | Role/task centric   | Very good          | Fast prototyping    |
| AutoGen     | Conversation hist. | External integration  | Message-based       | Good (group chat)  | Research / flexible |
| Mem0        | —              | Specialized long-term | Vector + graph      | Add-on             | Pure memory layer   |

### 6. Hands-on Starting Point (LangGraph + Redis)
Many 2026 production systems look like:
- LangGraph for orchestration & state
- Redis for <1 ms checkpoint + vector memory
- Background job for episodic summarization

Start small: Add `RunnableWithMessageHistory` to an existing chain → migrate to LangGraph when you need loops / branching / persistence.

# Episodic memory in agents

Here is a practical, step-by-step code walk-through for implementing **episodic memory** in a LangGraph-based agent (the dominant production pattern in 2025–2026).

Episodic memory in agents usually means:

- Recording specific past **experiences** (interactions, decisions, outcomes)
- Using them later as **few-shot examples** (to teach the agent correct behavior)
- Or as **summarized episodes** (to avoid blowing up the context window)

We'll build a realistic pattern used in many production systems:

1. Use **LangGraph checkpoints** for short-term + durable episodic traces (per thread/session)
2. After important steps or at session end, **summarize** raw history → episodic memory entries
3. Store episodic memories in a vector store (or simple list + similarity search)
4. At the beginning of new interactions, **retrieve relevant episodes** and inject them as few-shot examples or context

This is inspired by patterns from LangMem, DeepLearning.AI courses, and common GitHub repos (NirDiamant, YunghuiHsu forks, etc.).

### Full Walk-through Code

```python
import os
from datetime import datetime
from typing import Annotated, TypedDict, List

from langchain_core.messages import BaseMessage, AIMessage, HumanMessage
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_openai import ChatOpenAI

from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph.message import add_messages

# ────────────────────────────────────────────────
# 1. Define the persistent episodic memory store
#    (simple in-memory vector-like store for demo; use Pinecone/Chroma/FAISS in prod)
# ────────────────────────────────────────────────

class EpisodicMemoryEntry:
    def __init__(self, summary: str, timestamp: str, thread_id: str, outcome: str = ""):
        self.summary = summary
        self.timestamp = timestamp
        self.thread_id = thread_id
        self.outcome = outcome  # e.g. "success", "failed - timeout", "user corrected"

episodic_store: List[EpisodicMemoryEntry] = []   # In prod → vector DB + embeddings


def save_episodic_memory(summary: str, thread_id: str, outcome: str = ""):
    """Called after important episodes or session end"""
    entry = EpisodicMemoryEntry(
        summary=summary,
        timestamp=datetime.now().isoformat(),
        thread_id=thread_id,
        outcome=outcome
    )
    episodic_store.append(entry)
    print(f"[Episodic] Saved: {summary[:60]}...")


def retrieve_relevant_episodes(query: str, top_k: int = 2) -> List[str]:
    """
    Very naive similarity for demo.
    In production: embed query & entries → cosine similarity → top_k
    """
    # Dummy: just substring match or recency (replace with real vector search)
    relevant = []
    for entry in sorted(episodic_store, key=lambda x: x.timestamp, reverse=True)[:8]:
        if query.lower() in entry.summary.lower() or "error" in entry.outcome:
            relevant.append(f"Past episode ({entry.timestamp}): {entry.summary}")
            if len(relevant) >= top_k:
                break
    return relevant


# ────────────────────────────────────────────────
# 2. Agent State – includes short-term messages + episodic context
# ────────────────────────────────────────────────

class AgentState(TypedDict):
    messages: Annotated[List[BaseMessage], add_messages]
    # Episodic context injected at the start of reasoning
    episodic_context: List[str]


# ────────────────────────────────────────────────
# 3. Nodes
# ────────────────────────────────────────────────

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.3)

# Prompt that can consume episodic few-shot / examples
prompt = ChatPromptTemplate.from_messages([
    ("system", """You are a helpful assistant that learns from past experiences.
Use the provided episodic memories to avoid repeating mistakes and follow successful patterns.

{episodic_context}

Current task:"""),
    MessagesPlaceholder("messages"),
])

chain = prompt | llm


def agent_node(state: AgentState):
    # Inject episodic memory before calling LLM
    query = state["messages"][-1].content if state["messages"] else ""
    episodes = retrieve_relevant_episodes(query, top_k=2)

    response = chain.invoke({
        "messages": state["messages"],
        "episodic_context": "\n".join([f"- {e}" for e in episodes]) if episodes else "No relevant past episodes."
    })

    return {
        "messages": [response],
        # Optionally accumulate more episodic context here
    }


def summarize_and_save_episodic(state: AgentState, thread_id: str):
    """
    Reflection / summarization node.
    In production: run periodically or after critical steps.
    """
    if len(state["messages"]) < 3:
        return state  # too short

    summary_prompt = ChatPromptTemplate.from_messages([
        ("system", "Summarize the key events, decisions, and outcome of this conversation concisely."),
        MessagesPlaceholder("messages"),
    ])

    summary_chain = summary_prompt | llm
    summary = summary_chain.invoke({"messages": state["messages"]}).content

    # Determine rough outcome (heuristic – improve with classifier)
    outcome = "success"
    if any("error" in m.content.lower() for m in state["messages"][-4:]):
        outcome = "failed"

    save_episodic_memory(summary, thread_id, outcome)

    return state


# ────────────────────────────────────────────────
# 4. Build Graph
# ────────────────────────────────────────────────

workflow = StateGraph(state_schema=AgentState)

workflow.add_node("agent", agent_node)
workflow.add_node("summarizer", summarize_and_save_episodic)

workflow.add_edge(START, "agent")
workflow.add_conditional_edges(
    "agent",
    lambda s: "summarizer" if len(s["messages"]) % 5 == 0 else END,   # every 5 turns → summarize
    {"summarizer": "summarizer", END: END}
)
workflow.add_edge("summarizer", END)

# Persistent checkpoints (short-term + episodic trace durability)
checkpointer = MemorySaver()   # Swap to PostgresSaver / RedisSaver in prod

graph = workflow.compile(checkpointer=checkpointer)


# ────────────────────────────────────────────────
# 5. Run example sessions
# ────────────────────────────────────────────────

def run_conversation(thread_id: str, user_input: str):
    config = {"configurable": {"thread_id": thread_id}}

    # Before starting, we could also inject episodic at graph entry, but we do it inside agent

    events = graph.stream(
        {"messages": [HumanMessage(content=user_input)]},
        config,
        stream_mode="values"
    )

    for event in events:
        if "messages" in event and event["messages"]:
            last_msg = event["messages"][-1]
            if isinstance(last_msg, AIMessage):
                print(f"Agent: {last_msg.content[:120]}...")

    # Force episodic save at end of session (optional)
    current_state = graph.get_state(config).values
    summarize_and_save_episodic(current_state, thread_id)


# ────────────────────────────────
# Demo runs
# ────────────────────────────────

print("Session 1 – fails due to bad plan")
run_conversation("task-001-supplychain", "Plan how to restock 500 units of product XYZ when warehouse is in New Jersey but demand is highest in California.")

print("\nSession 2 – should learn from past mistake via episodic retrieval")
run_conversation("task-002-supplychain", "Plan restocking for product XYZ again – remember what went wrong last time.")
```

### What Happens Under the Hood

1. **Short-term episodic trace** → stored automatically via checkpoints (thread_id)
2. **Long-term episodic memory** → created by summarization node → saved to `episodic_store`
3. **Retrieval & injection** → at every agent call, relevant past episodes are pulled in as context / few-shot style guidance
4. In production you replace the dummy `retrieve_relevant_episodes` with:
   - Embedding model (OpenAI / sentence-transformers)
   - Vector DB query (FAISS, Chroma, Pinecone, Weaviate)
   - MMR reranking for diversity

### Production Hardening Tips (2026 style)

- Use **LangMem** `Memory` abstraction for managed episodic + semantic layers
- Add **importance scoring** before saving (LLM judge: 1–10 usefulness)
- Run summarization **async/background** after session
- TTL + deduplication on episodic store
- Human feedback loop: allow users to label episodes (“this worked well”)

This pattern gives your agent the ability to **remember specific experiences** and **improve behavior over time** — exactly what episodic memory is meant for.
